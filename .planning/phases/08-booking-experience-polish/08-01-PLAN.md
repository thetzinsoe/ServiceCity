# Plan 08-01: Booking Form Autofill

## Objective

Pre-fill the booking form's identity fields (name, phone, address) from the registered customer's account so they don't re-enter the same information every time they book.

Purpose: Reduce friction for repeat customers. They already told us who they are during registration ŌĆö no need to type it again.

Output: BookingController.Create GET loads the authenticated user and populates BookingViewModel. The form renders with pre-filled values. All fields remain editable.

Add `using System.Security.Claims;` at the top if not already present (it's already used in Create POST).

Replace the Create GET action body. After the existing ServiceCategory lookup and before `return View(...)`:

1. Extract userId from claims:
   ```
   var userIdClaim = User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
   ```

2. If userId exists and parses to int, load the User:
   ```
   if (userIdClaim != null && int.TryParse(userIdClaim, out var userId))
   {
       var user = await db.Users.FindAsync(userId);
       if (user != null)
       {
           model.CustomerName = user.Name;
           model.CustomerPhone = user.PhoneNumber;
           model.Address = user.Address;
       }
   }
   ```

3. The model still starts with `ServiceCategoryId` set from the route param. The identity fields get populated from the User entity. `Description`, `PreferredDate` (defaults to tomorrow), and `PreferredTimeSlot` stay at their defaults ŌĆö never pre-filled per D-02.

**No changes needed to Create POST** ŌĆö it already handles the submitted values correctly, normalizes the phone, and sets `booking.UserId` from claims. The pre-filled values flow through the existing form submission pipeline unchanged.

**No changes to BookingViewModel** ŌĆö it already has `CustomerName`, `CustomerPhone`, and `Address` properties with the right validation attributes. Pre-filling them server-side works with the existing model binding.
