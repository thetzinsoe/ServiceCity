    **Part B — AuthController additions (append to existing AuthController.cs):**

    Open ServiceCity/Controllers/AuthController.cs. The file already contains Setup GET, Setup POST, and ValidateAndNormalizePhone from Plan 02-01. Add the following new actions AFTER the existing members but INSIDE the closing brace of the class. Also ensure the `using Microsoft.AspNetCore.Authentication;` and `using Microsoft.AspNetCore.Authentication.Cookies;` and `using System.Security.Claims;` and `using Microsoft.AspNetCore.Identity;` statements are present at the top of the file; add any that are missing.

    1. **SignIn GET** — `[HttpGet] public IActionResult SignIn()`: Return View().

    2. **SignIn POST** — `[HttpPost] public async Task<IActionResult> SignIn(SignInViewModel model)`:
       - Check `ModelState.IsValid` — if false, return View(model).
       - Query: `var user = await db.Users.FirstOrDefaultAsync(u => u.Username == model.Username)`.
       - If user is null OR user.PasswordHash is null/empty — add ModelState error with key string.Empty and message "Invalid username or password." Return View(model).
       - Instantiate `var hasher = new PasswordHasher<User>()`.
       - Call `var result = hasher.VerifyHashedPassword(user, user.PasswordHash, model.Password)`.
       - If result == PasswordVerificationResult.Failed — add ModelState error with key string.Empty and message "Invalid username or password." Return View(model).
       - For both failure cases above, use the exact same error message and key string.Empty — per Claude's Discretion, never reveal which field was wrong (prevents username enumeration per RESEARCH.md Security Domain).
       - Build ClaimsPrincipal: create `new List<Claim>` with ClaimTypes.NameIdentifier = user.Id.ToString(), ClaimTypes.Name = user.Username, ClaimTypes.Role = "Admin".
       - Create `new ClaimsIdentity(claims, CookieAuthenticationDefaults.AuthenticationScheme)`.
       - Create `new ClaimsPrincipal(identity)`.
       - Call `await HttpContext.SignInAsync(CookieAuthenticationDefaults.AuthenticationScheme, principal)`.
       - Return `RedirectToAction("Index", "Admin")` — per Claude's Discretion, redirect to admin dashboard after sign-in.

    3. **SignOut POST** — `[HttpPost] public async Task<IActionResult> SignOut()`:
       - Call `await HttpContext.SignOutAsync(CookieAuthenticationDefaults.AuthenticationScheme)`.
       - Return `RedirectToAction("SignIn")` — redirects to the sign-in page after sign-out.

    **Key implementation notes for executor:**
    - Use `PasswordVerificationResult.Failed` (not != Success — the enum has Success, Failed, SuccessRehashNeeded). RESEARCH.md Pattern 2 shows `== PasswordVerificationResult.Failed` for the rejection condition. If the result is SuccessRehashNeeded, the sign-in still succeeds (password correct, hash format upgradeable later).
    - Do NOT use `UserManager<T>` or any Identity abstractions — this is cookie auth without ASP.NET Core Identity.
    - The `db` field is already available via primary constructor DI from Plan 02-01.
    - The `using` statements needed: `using Microsoft.AspNetCore.Authentication;`, `using Microsoft.AspNetCore.Authentication.Cookies;`, `using System.Security.Claims;`, `using Microsoft.AspNetCore.Identity;`, `using PhoneNumbers;` (already present from Plan 02-01).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Server (SignIn form) | Untrusted input: Username, Password — user-supplied credentials |
| Client -> Server (Cookie) | Auth cookie presented on every request — must be HttpOnly to block XSS theft |
| Server -> AdminController | [Authorize] gate — unauthenticated requests redirected to LoginPath |
| SignOut route | State-changing action — must be POST-only to prevent CSRF |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Information Disclosure | AuthController.SignIn POST | mitigate | Generic "Invalid username or password." message for both wrong username and wrong password. Identical error key (string.Empty). No timing differentiation. Per RESEARCH.md Security Domain. |
| T-02-09 | Spoofing | Cookie authentication | mitigate | ASP.NET Core regenerates cookie value on each SignInAsync call. Old session values invalidated. Cookie configured HttpOnly=true (prevents JS access), SameSite=Lax (prevents cross-site cookie sending). |
| T-02-10 | Tampering | SignOut action | mitigate | SignOut is POST-only via `<form method="post">` with `<button type="submit">` — CSRF via `<img>` or `<a>` tag is not possible. ASP.NET Core antiforgery auto-validates on POST. |
| T-02-11 | Information Disclosure | Auth cookie in transit | mitigate | Cookie configured with HttpOnly=true. Secure=SameAsRequest (auto-enables Secure flag when HTTPS is active). Cookie encrypted via ASP.NET Core Data Protection (AES-256-CBC + HMACSHA256). |
| T-02-12 | Elevation of Privilege | AdminController | mitigate | All actions protected by class-level `[Authorize]` attribute. Unauthenticated requests automatically redirected to LoginPath=/Auth/SignIn. AccessDeniedPath also set to /Auth/SignIn. |
| T-02-13 | Repudiation | Sign-in/sign-out events | accept | No audit log for auth events in v1. Single admin shop — accountability is inherent. Audit logging can be added later without schema changes. |
| T-02-14 | Denial of Service | AuthController.SignIn POST | accept | Brute-force on sign-in endpoint. PBKDF2 hashing (~100ms per attempt) provides some resistance. Rate limiting on auth routes is deferred to Phase 6 (CROS-05). Single admin reduces attack surface. |
</threat_model>
