# Phase 09: Layered Architecture Refactor — Controller → Service → Repository

**Branch:** `refactor/layered-architecture`
**Status:** Planning
**Date:** 2026-06-21

---

## Architecture Decision Record

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Service project** | New `ServiceCity.Services` project | DLL boundary enforces interface discipline, enables unit testing |
| **Migration strategy** | All-at-once | Single branch, consistent — avoid half-migrated state |
| **DTOs + ViewModels** | Both, separate | DTOs carry data between layers, ViewModels are Razor-display-only |
| **Repository location** | Inside `ServiceCity.Data` | Repositories are data access — they belong with DbContext |
| **Phone validation** | Shared `IPhoneValidationService` | Eliminates duplication across AuthController + BookingController |

## Data Flow

```
┌─ GET Request ──────────────────────────────────────────────────┐
│                                                                 │
│  Browser ──► Controller ──► IService.GetXxx(params)             │
│                        │        │                               │
│                        │        ├─ IRepository.Query/Find       │
│                        │        ├─ Business logic               │
│                        │        └─ Returns DTO                  │
│                        │                                       │
│                        │  Map DTO → ViewModel                   │
│                        │                                       │
│                        └─► View(ViewModel) ──► Browser          │
└─────────────────────────────────────────────────────────────────┘

┌─ POST Request ─────────────────────────────────────────────────┐
│                                                                 │
│  Browser ──► Controller(model-binds Request)                    │
│                        │                                        │
│                        │  Validate, then:                       │
│                        ├─► IService.DoXxx(Request, userId)      │
│                        │        │                                │
│                        │        ├─ IRepository.Get/Add/Update   │
│                        │        ├─ Business rules               │
│                        │        └─ Returns DTO                  │
│                        │                                        │
│                        └─► Redirect / View(ViewModel)           │
└─────────────────────────────────────────────────────────────────┘
```

### Task 1: Create Repository layer (in ServiceCity.Data)

**Repository interfaces + implementations** — each wraps EF Core operations:

| Repository | Methods |
|-----------|---------|
| `IBookingRepository` | `GetAllAsync()`, `GetByIdAsync(id)`, `GetByReferenceAsync(ref)`, `FindAsync(id)`, `AddAsync(booking)`, `Query()` (return IQueryable), `SearchAsync(term)`, `GetByUserIdAsync(userId)`, `GetCountsByStatusAsync()` |
| `IUserRepository` | `FindAsync(id)`, `GetByPhoneAsync(normalized)`, `GetByUsernameAsync(username)`, `GetNonAdminQuery()`, `SearchAsync(term)` |
| `INotificationRepository` | `AddAsync(notification)` |
| `IServiceCategoryRepository` | `GetAllAsync()`, `FindAsync(id)` |

- Repositories expose `Query()` returning `IQueryable<T>` for composable filtering (with `.AsNoTracking()` when reading)
- All write methods call `SaveChangesAsync()` internally
- Each repository injects `AppDbContext` via primary constructor

### Task 4: Create Service interfaces + implementations

**Service contracts:**

| Interface | Methods |
|-----------|---------|
| `IBookingService` | `CreateBookingAsync(CreateBookingRequest, userId?)` → `BookingDto`, `GetConfirmationAsync(refNumber)` → `BookingDto`, `GetStatusAsync(refNumber)` → `BookingDto`, `GetMyBookingsAsync(userId)` → `List<BookingListItemDto>`, `GenerateReferenceNumber()` → `string` |
| `IAuthService` | `SignInAsync(SignInRequest)` → `(UserDto?, string? error)`, `RegisterAsync(RegisterRequest)` → `(UserDto?, string? error)`, `SetupAsync(SetupRequest)` → `(UserDto?, string? error)`, `GetClaimsPrincipal(user)` → `ClaimsPrincipal`, `UpdateSettingsAsync(userId, SettingsRequest)` → `(UserDto?, string? error)` |
| `IAdminService` | `GetDashboardAsync(search, status)` → `AdminDashboardDto`, `GetDrilldownAsync(status, search)` → `AdminDashboardDto`, `GetBookingDetailAsync(id)` → `BookingDetailDto`, `AcceptAsync(id, arrivalTime)`, `DeclineAsync(id, reason)`, `StartInProgressAsync(id)`, `CompleteAsync(id)`, `GetCustomersAsync(search)` → `List<CustomerListItemDto>`, `GetCustomerDetailAsync(id)` → `(UserDto, List<BookingListItemDto>)` |

**Services contain all business logic:**
- `AuthService` — password hashing (PasswordHasher), claims construction, sign-in via HttpContext (or return claims principal for controller to sign in)
- `BookingService` — idempotency check, reference number generation, notification creation on booking
- `AdminService` — status transition validation (can't accept non-pending, etc.), notification messages
- `PhoneValidationService` — shared phone validation

### Task 6: Refactor controllers (thin them down)

**Each controller becomes: inject services → validate → call service → map DTO→ViewModel → return view.**

Example — `AuthController` after refactor:
```csharp
public class AuthController(IAuthService auth, IPhoneValidationService phoneValidator) : Controller
{
    [HttpPost]
    public async Task<IActionResult> SignIn(SignInRequest model)
    {
        if (!ModelState.IsValid) return View(model);
        
        var (user, error) = await auth.SignInAsync(model);
        if (error != null)
        {
            ModelState.AddModelError(string.Empty, error);
            return View(model);
        }
        
        // Auth service returns ClaimsPrincipal, controller signs in
        await HttpContext.SignInAsync(principal);
        return user.IsAdmin 
            ? RedirectToAction("Dashboard", "Admin") 
            : RedirectToAction("Index", "Home");
    }
}
```

**Lines of code impact per controller:**

| Controller | Before | After (estimated) |
|-----------|--------|-------------------|
| `AuthController` | 270 lines | ~80 lines |
| `AdminController` | 223 lines | ~60 lines |
| `BookingController` | 208 lines | ~65 lines |
| `HomeController` | 29 lines | ~15 lines |
| **Total** | **730 lines** | **~220 lines** |

Controllers go from ~730 lines to ~220 — a 70% reduction.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Build break during incremental refactor | All-at-once approach: complete all layers before building |
| DI registration miss | Each interface has a single implementation — easy to verify in Program.cs |
| Property name mismatch after renaming ViewModels→Requests | Search-and-replace across .cshtml files for `asp-for` tags |
| EF Core query differences after repository wrapper | Repositories return IQueryable for composable queries; SaveChangesAsync handled by repository |
